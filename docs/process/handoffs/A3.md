# A3 — Persistence & Migrations (Handoff)

- **Owning agent:** A3 · **Owns:** `backend/prisma/**`, `backend/src/db/**`
- **Source of truth:** `backend/prisma/schema.prisma`
- **Related:** ADR-0002 (DB client contract), ADR-0007 (`DATABASE_URL`), TDR (Prisma 5, PostgreSQL 16), `contracts/openapi.yaml`, `docs/architecture/database-overview.md`

## What shipped
- `backend/prisma/schema.prisma` — datasource `postgresql` (`url = env("DATABASE_URL")`), `citext` extension enabled via the `postgresqlExtensions` preview feature. Entities: `User`, `VerificationToken`, `Team`, `Epic`, `Ticket`, `Comment`. All PKs are `uuid` (`@default(uuid())` / `@db.Uuid`). Enums `TicketType` and `TicketState` match the OpenAPI contract exactly.
- `backend/prisma/migrations/20260707000000_init/migration.sql` — initial migration generated offline (`prisma migrate diff --from-empty --to-schema-datamodel`). Creates the `citext` extension, both enums, all six tables, unique/secondary indexes, and all foreign keys. **Contains zero application rows — no seed data.**
- `backend/prisma/migrations/migration_lock.toml` — provider `postgresql`.
- `backend/src/db/client.ts` — `export const prisma` singleton.
- `backend/src/db/index.ts` — public entrypoint: re-exports `{ prisma }`, plus `getDb()` and `dbHealthcheck()`.

## Database Client Contract (restated — FROZEN, ADR-0002)
Consumers **A4, A5, A6, A7, A8, A15** access the DB through one singleton:

```ts
import { prisma } from 'backend/src/db';
```

- `export const prisma` — the single `PrismaClient` instance, defined in `backend/src/db/client.ts`.
- `backend/src/db/index.ts` is the **only** public entrypoint; it re-exports `{ prisma }`.
- **No consumer instantiates its own `PrismaClient`** (one process → one connection pool).
- The `prisma` export signature is frozen; changing it requires a superseding ADR (MO-ratified, G1-logged).
- Convenience helpers `getDb()` (returns the singleton) and `dbHealthcheck()` (runs `SELECT 1`, resolves `true` or throws) are provided for A4's `/ready` probe. Per ADR-0002 these are **not** part of the frozen signature — the only guaranteed, frozen surface is `prisma`. Treat them as convenience; the liveness query itself is the contract, and A4 may equally call `prisma.$queryRaw` directly.

## Case-insensitive uniqueness — mechanism chosen
**PostgreSQL `citext` extension** (not a normalized lowercase column).
- `User.email` and `Team.name` are typed `@db.Citext` with `@unique`.
- The migration runs `CREATE EXTENSION IF NOT EXISTS "citext";` before the tables.
- Enabled in Prisma via `generator.previewFeatures = ["postgresqlExtensions"]` + `datasource.extensions = [citext]`.
- Effect: `alice@x.com` and `Alice@X.com` collide on the unique index; equality/`WHERE` comparisons are case-insensitive automatically — services do **not** need to lowercase before querying. (Trim input at the service layer per the contract; citext handles case only.)

## Constraint → HTTP status mapping (for A5–A8)
On a write, catch the Prisma error and translate. Prisma surfaces DB constraint violations as `PrismaClientKnownRequestError` with a `code`:

| DB constraint | Prisma code | HTTP | Error `code` (OpenAPI) |
|---|---|---|---|
| `users_email_key` (unique email) | `P2002` | 409 | `EMAIL_TAKEN` |
| `teams_name_key` (unique name) | `P2002` | 409 | `TEAM_NAME_TAKEN` |
| Delete `Team` referenced by epics/tickets (FK RESTRICT) | `P2003` (or `P2014`) | 409 | `TEAM_HAS_DEPENDENTS` |
| Delete `Epic` referenced by tickets (FK RESTRICT) | `P2003` (or `P2014`) | 409 | `EPIC_REFERENCED` |
| Insert/update FK to a missing `Team`/`Epic`/`User` | `P2003` | 404 | `TEAM_NOT_FOUND` / `NOT_FOUND` |
| Row not found on get/update/delete | `P2025` | 404 | `NOT_FOUND` |

Notes for feature agents:
- Disambiguate `P2002` by inspecting `err.meta.target` (contains the failing index/columns) to pick `EMAIL_TAKEN` vs `TEAM_NAME_TAKEN`.
- Disambiguate a RESTRICT delete (`P2003`) by which entity is being deleted (team → `TEAM_HAS_DEPENDENTS`, epic → `EPIC_REFERENCED`).
- **Cross-team epic** (`EPIC_TEAM_MISMATCH`, 400) is **not** a DB constraint — there is no DB rule that an epic's team must equal the ticket's team. Enforce it in the service layer on ticket create/update, as the contract specifies.
- `ticket.type` / `ticket.state` invalid enum values are rejected by request validation (400 `VALIDATION_ERROR`) before hitting the DB.

## FK referential actions (as migrated)
| Relation | onDelete |
|---|---|

| `VerificationToken.user` → `User` | CASCADE |
| `Epic.team` → `Team` | **RESTRICT** → 409 |
| `Ticket.team` → `Team` | **RESTRICT** → 409 |

## Field / column naming
Model fields are camelCase and match the OpenAPI schema field names (`teamId`, `epicId`, `createdBy`, `authorId`, `ticketId`, `createdAt`, `modifiedAt`). Physical columns/tables are snake_case (`created_by`, `verification_tokens`, …) via `@map`/`@@map`. Timestamps are `timestamptz(3)`, UTC, server-set; `modifiedAt` uses `@updatedAt`.

## Migrations & seed policy
Applied on boot via `npm run migrate` (`prisma migrate deploy`, script owned by A4, triggered by A2's container entrypoint). Fresh DB → schema + `_prisma_migrations` metadata only. **No seed / sample data** on any default path.

## Verification performed
- `prisma validate` (Prisma 5.22.0) → schema valid.
- `prisma format` → clean.
- `prisma migrate diff --from-empty` → migration generated offline, includes `CREATE EXTENSION citext`, enums, tables, indexes, FKs; no `INSERT` statements (zero app rows).
- **Not run:** live apply against a real Postgres and runtime constraint exercise — this environment has no Docker/Postgres. The migration was authored via the offline diff path (the charter's sanctioned no-live-DB option). First `migrate deploy` on the compose Postgres (A2) is the final apply check.
