## Cross-feature contract — `ticketRoutes` (ADR-0008, FROZEN)

Exported from **`@/features/tickets/routes.tsx`** (re-exported from `paths.ts` to avoid an import
cycle through the page components), exactly per ADR-0008:

```ts
ticketRoutes.create()      // '/tickets/new'
ticketRoutes.detail(id)    // `/tickets/${id}`
ticketRoutes.edit(id)      // `/tickets/${id}/edit`
```

`routes: RouteObject[]` is also exported and auto-discovered by `@/app/router`'s
`import.meta.glob('../features/*/routes.tsx', …)` — no `@/app` edit. All three ticket screens are
wrapped in `<RequireAuth>`.

**A12:** navigate via `import { ticketRoutes } from '@/features/tickets/routes'`. Do not hard-code
ticket URLs.

## Behavior notes

- **All fields shown.** View renders id, title, type, state, team, epic, body, and the read-only
  `createdBy` / `createdAt` / `modifiedAt`. Editable fields (form): type, team, epic, title, body,
  state — matching §6.
- **Team change clears epic.** `TicketForm.handleTeamChange` sets the new `teamId` and resets
  `epicId` to `null`; the epic dropdown reloads because `useTeamEpics` is keyed by `teamId`
  (`enabled: !!teamId`). Backend also rejects cross-team epics (EPIC_TEAM_MISMATCH) — surfaced via
  the mutation error Toast.
- **Delete confirmation.** Explicit `<Modal>` confirm ("Delete + all comments"); on 204 success the
  user is navigated to the board (`BOARD_PATH = '/board'`, per A9's example and §8).
- **Comments oldest-first.** The API returns chronological order; the panel renders as-received.
  Add-comment requires a non-empty (trimmed) body; adding a comment invalidates only the comments
  list (ticket `modifiedAt` is unaffected — server-enforced, §7).
- **State/type labels.** Canonical API enum values (`@app/shared` `TICKET_STATES`/`TICKET_TYPES`)
  are rendered via human-readable label maps.
- **Foundation states** (`LoadingState`/`EmptyState`/`ErrorState`) drive fetch UI; `Toast` surfaces
  mutation errors. Uses only the frozen `@/components` set + native `<select>`/`<textarea>` styled
  with the `ds-input` class (no additions to `@/components`).

## Design decisions / advisories

- **Comments embedded via a feature barrel.** `TicketView` imports `CommentsPanel` from
  `@/features/comments`. The comments panel is an embedded component (not a separate screen), and
  the route-registry composes features as a flat list (no cross-feature `<Outlet>` nesting), so
  router composition is not possible. Both features are the single A13 Phase-1 deliverable;
  `@/features/comments/index.ts` is the comments feature's published entry point. This is the only
  cross-feature import between the two A13 features.
- **Contract types from `@app/shared`.** Domain types (`components['schemas'][...]`) and the enum
  constants/arrays are imported from the shared contract package (the FE+BE source of truth that
  `@/lib/api` itself uses) — not from any other feature. No second API/query client is created.
- **No new dependencies** (TDR frozen): React, React Router 6, TanStack Query 5, openapi-fetch,
  `@app/shared` only.

## Notes for downstream

- The board redirect target is `'/board'` (A9's example path). If A12 registers the board under a
  different path, update `BOARD_PATH` in `frontend/src/features/tickets/paths.ts` (single-valued).
- `@/lib/api` does not currently attach the `Authorization` bearer header (A9/A10 foundation
  concern, out of A13 scope). Ticket/comment requests assume the foundation supplies auth; no
  workaround was added in the feature layer.
