## Design decisions

- **Server-side enum + shape validation** is enforced by Fastify/Ajv via the `@app/shared` schema
  attached per route (invalid `type`/`state` → `400 VALIDATION_ERROR` before the handler runs).
  No enum list is re-declared in this module (ADR-0006 forbids it).
- **`createdBy` / `createdAt` are server-set:** `createdBy = request.authUser.id` (ADR-0004);
  `createdAt` is the DB `@default(now())`. The request body schema (`additionalProperties: false`)
  cannot even carry these.
- **Epic-team invariant (§5/§6)** is checked in the service on create AND update via
  `validateReferences(teamId, epicId)`: team must exist (else `404 TEAM_NOT_FOUND`), epic must exist
  (else `404 EPIC_NOT_FOUND`) and share the ticket's team (else
  `throw new ValidationError('EPIC_TEAM_MISMATCH', …)` → 400). On update, the *effective* post-update
  team/epic are validated, so a team change that would orphan the current epic is rejected — matching
  the contract rule that changing team requires a compatible `epicId` or `epicId: null`.
  This is a service-layer rule (no DB constraint exists for it) per A3's handoff.
- **`modifiedAt` advances only on a real change (§6):** the schema uses `@updatedAt`, which bumps
  `modifiedAt` on *every* `update`. So `updateTicket` dirty-checks each field against the current row
  and, when nothing actually differs, **returns the current row without issuing a write** — a no-op
  save therefore preserves `modifiedAt`. Only genuinely changed columns are sent in `data`.
- **Delete cascade:** `prisma.ticket.delete` relies on the `Comment.ticket` FK `ON DELETE CASCADE`
  (A3) to remove comments; a missing row (`P2025`) maps to `404 TICKET_NOT_FOUND`.
- **Board query (§8):** `where` is `{ teamId, type?, epicId?, title.contains(q, mode:'insensitive') }`
  (AND-combined), `orderBy: { modifiedAt: 'desc' }`. The `teamId` filter is served by A3's
  `tickets_team_id_idx`; at ~100 rows/team the subsequent in-memory sort is negligible.

## Coordination with A8 (comments) — modifiedAt

Adding a comment MUST NOT touch `ticket.modifiedAt` (§6/§7). This module never provides a
comment-write path, and A8's comment creation writes to the `comments` table only (never calls
`prisma.ticket.update`), so `modifiedAt` is structurally untouched by comments. **A8: do not call
into the tickets service and do not update the ticket row when appending a comment.**

## Phase 3 — R4 red-team M1 fix (title/body trim-and-reject)

- **Finding (R4-M1, Medium, §6 conformance):** ticket `title`/`body` were stored raw. A whitespace-only
  value (e.g. `"   "`) passed Ajv `minLength:1` and persisted, violating §6 "Title required, non-empty
  after trim" / "Body required, non-empty". Applied to both create and PATCH.
- **File changed:** `backend/src/modules/tickets/service.ts` (only). No contract, response-shape, or
  enum/state changes.
- **Fix:** added a `normalizeText(raw, field)` helper mirroring Teams `normalizeName` / Epics
  `normalizeTitle` — it `.trim()`s and, when the result is empty, throws
  `new ValidationError('VALIDATION_ERROR', …)` from `backend/src/core/errors` (ADR-0003 → 400 via A4's
  handler). No new error code invented.
  - **`createTicket`:** `title`/`body` are now `normalizeText(…)`'d before persist (trimmed value stored).
  - **`updateTicket`:** when `title`/`body` are present they are `normalizeText(…)`'d *before* the
    dirty-check, so (a) whitespace-only → 400, (b) the trimmed value is stored, (c) a value that trims to
    the current stored value is still a no-op (modifiedAt preserved), (d) an absent field is untouched.
- **Fields trimmed:** `title`, `body` (the only §6 free-text non-empty fields; `type`/`state` are enums,
  refs are IDs).
- **Error path:** `service.ts normalizeText` → `throw new ValidationError('VALIDATION_ERROR', …)` →
  A4 `registerErrorHandler` → `400 { code: 'VALIDATION_ERROR', message }`.
- **Verification:** `tsc --noEmit` clean; existing `tests/unit/tickets-service.test.ts` 11/11 pass
  (modifiedAt no-op + valid behavior unchanged).
