## Routes registered (ADR-0008 route-registry)

Each feature exports `routes: RouteObject[]` from `routes.tsx`, auto-discovered by
`@/app/router`'s `import.meta.glob('../features/*/routes.tsx')`. No `@/app` edit was made.

- `/teams` → `<RequireAuth><TeamsPage/></RequireAuth>` (protected)
- `/epics` → `<RequireAuth><EpicsPage/></RequireAuth>` (protected)

Both pages render a small in-page nav (`Board` / `Teams` / `Epics`) so users can move between the
management screens and the board without any header edit (the shell header is A9-owned).

  `/epics`, `/epics/{epicId}`, and read-only `GET /tickets` / `GET /epics` for reference detection.
- `@/lib/auth` — `RequireAuth` guard on both routes.
- `@/components` — `Button, Input, Field, Modal, Toast, LoadingState, EmptyState, ErrorState`.

## Behavior

**Teams (§4):** list (ordered by name from the API), create, rename, delete. Create/rename go
through a modal; `name` is trimmed and required client-side; a backend 409 `TEAM_NAME_TAKEN` is
surfaced inline as the field error. Delete is behind a confirmation modal.

**Epics (§5):** list (optionally filtered by a team selector), create, edit, delete. **Team is
chosen only at creation** (dropdown in the create modal) and is **immutable** afterward — the edit
modal shows the team as a read-only value and `updateEpic` sends only `title`/`description`
(matching `EpicUpdateRequest`, which has no `teamId`). Description is optional (sent as `null` when
blank). "New epic" is disabled with a tooltip when no teams exist.

**Delete-disabled-when-referenced (acceptance):** there is no count field on `Team`/`Epic`, so
reference state is derived from the API:
- A **team** is referenced (delete disabled) if it has ≥1 epic (from `GET /epics`) OR ≥1 ticket
  (`GET /tickets?teamId=`). Tooltip: *"This team has tickets or epics and cannot be deleted."*
- An **epic** is referenced (delete disabled) if any ticket points at it
  (`GET /tickets?teamId=&epicId=`). Tooltip: *"This epic is referenced by tickets and cannot be
  deleted."*
- While reference state is still loading, delete is disabled defensively with a "Checking…" tooltip.
- The tooltip is carried on a wrapper `<span title="…">` so it appears even over a disabled button
  (`@/components` `Button` has no `title` prop; adding one would need a superseding ADR).

**409 surfaced clearly (acceptance):** delete mutations catch the error and show a fixed-position
error `Toast` with the backend's exact message (`TEAM_HAS_DEPENDENTS` / `EPIC_REFERENCED`), so the
proactive disable and the authoritative backend guard both hold (last-write-wins race safe).

## Query-cache coherence (intentional)

Query keys are shared *by prefix* with the sibling feature on the single app `QueryClient`:
the teams page reads all epics under `['epics']`; the epics feature invalidates `['epics']` after
any create/update/delete. So creating an epic immediately makes its team undeletable on the Teams
screen, and deleting the last epic re-enables it — no manual coordination between the features.

## Notes / advisories

- **Nav to `/board`:** both pages link to `/board` (A12's route). Until A12 ships that route, the
  link lands on the foundation NotFound `EmptyState` — expected, resolves when A12 registers it.
- **Reference detection cost:** teams issue one `GET /tickets?teamId=` per team and epics one
  `GET /tickets?teamId=&epicId=` per epic (via `useQueries`) because `listTickets` requires a
  `teamId`. Fine for hackathon scale; if team/epic counts grow, a backend `hasDependents`/count
  field would collapse these to a single call (would need a contract change — out of A11 scope).
- **CSS duplication:** `teams.css` and `epics.css` are byte-identical (`.mng-*` classes). They are
  duplicated rather than shared because importing another feature's file is a forbidden deep import
  (ADR-0008). The global selectors are identical, so double-loading is harmless.
- **Cosmetic:** the "CJS build of Vite's Node API is deprecated" warning is A9's known harmless
  notice; build unaffected.
