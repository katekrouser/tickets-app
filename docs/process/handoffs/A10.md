## Routes contributed (ADR-0008 route-registry)

All **public** (no `<RequireAuth>`) — these are the auth flows themselves, the only screens exempt from
auth (REQUIREMENTS §3). They render inside `<AppShell>`; the header user menu auto-hides when logged out.

```
/login                 → LoginPage
/signup                → SignupPage
/verify-email          → VerifyEmailPage   (reads ?token=…)
/resend-verification   → ResendVerificationPage   (reads ?email=… to prefill)
```

A9's `import.meta.glob('../features/*/routes.tsx')` picks these up automatically. Build confirms the
module count rose from 85 (Phase 0) to 129, i.e. the auth feature composed into the router with no
`@/app` change.

## Endpoint mapping (contracts/openapi.yaml → tag `auth`)

| UI action | Call via `@/lib/api` | Handled responses |
|---|---|---|
| Sign up | `POST /auth/signup` | 201 → "check your email"; 409 `EMAIL_TAKEN` → email field error; 400 → form error |
| Log in | `POST /auth/login` | 200 → `useAuth().login(token)` + redirect; 401 `INVALID_CREDENTIALS`; 403 `EMAIL_NOT_VERIFIED` → inline resend |
| Verify email | `GET /auth/verify-email?token=` | 302→/login (followed) treated as success; 400 `TOKEN_EXPIRED\|TOKEN_USED\|TOKEN_INVALID` |
| Resend | `POST /auth/resend-verification` | 202 → generic "if an account exists…" (anti-enumeration); 400 → form error |

## Key design decisions

- **No auto-login post-verify (REQUIREMENTS §3).** `VerifyEmailPage` never imports or calls
  `useAuth().login()`. On success it shows a confirmation, then `navigate('/login', { replace: true })`
  (auto after ~2.5s + a manual "Continue to sign in" button). Verified: the only "login" token in that
  file is prose in a comment.
- **Verify-email redirect handling.** Per ADR-0009 §12 the backend answers `GET /auth/verify-email`
  with a `302 → ${APP_BASE_URL}/login`. Browser `fetch` transparently follows that same-origin redirect
  to the SPA's `index.html` (a 200 HTML body). The service therefore requests `parseAs: 'text'` and
  treats **"no error envelope" as success** — it never JSON-parses the redirected HTML. A `400` still
  surfaces the `TOKEN_*` code.
- **Single-use token safety.** The verification effect is guarded by a `useRef` so React StrictMode's
  double-invoked effect (dev only) cannot spend the single-use token twice (which would flip a valid
  token to `TOKEN_USED`).
- **Server-authoritative validation.** `validation.ts` mirrors the contract for fast feedback only;
  every submit is re-validated by the backend and its response is the source of truth. Emails are
  `.trim()`-ed before send (server compares case-insensitively).
- **Resend reachable from login + verification + signup** (REQUIREMENTS §3/§10): inline button on
  Login's 403 path, footer link on the verification screen, and a "Didn't get the email?" link on the
  sign-up success panel — all in addition to the standalone `/resend-verification` screen.
- **Container/presenter split** in `VerifyEmailPage`: the pure `VerifyEmailView(status)` presenter is
  exported so every outcome renders in isolation (used by the render verification below).

## Boundary note for A5 / integration

The emailed verification link is owned by **A5** and, per ADR-0009 §12, points at the backend
(`${APP_BASE_URL}/api/auth/verify-email?token=…`); the backend 302-redirects to `/login` on success.
This SPA `/verify-email` screen handles the **client-driven** verification path (link → SPA route with
`?token=`) so every result state has real UI. Either target satisfies REQUIREMENTS §3: success ends on
`/login` with **no** session created without an explicit sign-in. If the deployed email points the user
straight at the backend endpoint, success still lands on `/login` and only failures (raw 400 JSON) lack
the SPA's styled error panel — routing the link through `/verify-email?token=…` gives the full UX.

## Notes / advisories for downstream agents

- **Redirect-back after login.** `LoginPage` reads `location.state.from.pathname` (set by A9's
  `RequireAuth`) and returns there post-login, else `/`. Protected features get correct return-to.
- **Already-authenticated users** hitting `/login` are redirected to `/`.
- The auth service (`service.ts`) is the single seam onto `@/lib/api` for auth; other features should
  not import it (it's feature-internal, not a published cross-feature boundary).
