## How the frozen bare specifiers resolve (compile + BOTH runtimes)
The ADR specifiers `backend/src/db` (ADR-0002), `backend/src/core/errors` (ADR-0003), `backend/src/core/auth` (ADR-0004):
- **Compile time:** tsconfig `baseUrl: "." + paths: { "backend/src/*": ["./src/*"] }` under `moduleResolution: Bundler` (resolves both file targets, e.g. `core/errors`, and directory targets, e.g. `db/`).

- **Prod runtime (`node dist/server.js`):** `tsc` leaves the bare specifiers in the emitted JS; **`tsc-alias --resolveFullPaths`** rewrites them to correct relative ESM paths with explicit extensions/index (e.g. `backend/src/db` → `./db/index.js`, `backend/src/core/errors` → `./core/errors.js`). Verified booting from `dist/`.

## Error contract convention (ADR-0003)
- Feature modules `import { NotFoundError, ConflictError, ValidationError, ... } from 'backend/src/core/errors'` and **throw** them; never set status manually.
- `registerErrorHandler(app)` (called first in `buildApp`) maps: `DomainError` → `status` + `toBody()`; Fastify/Ajv validation or any malformed 400 → `400 { code: 'VALIDATION_ERROR', message, details.validation }`; other framework 4xx → preserved status + safe body; anything else → `500 { code: 'INTERNAL', message: 'Internal Server Error' }` (no stack/detail leak). A `notFoundHandler` returns `404 { code: 'NOT_FOUND', ... }`.
- `ErrorBody { code, message, details? }` matches the OpenAPI `Error` schema.

## Request-validation convention (ADR-0006)
Fastify's built-in Ajv (no extra validator). Each route attaches its schema from `@app/shared`:
```ts
import { schemas } from '@app/shared';
app.post('/', { schema: schemas['createTicket'] }, handler);
```
Feature modules apply this per-route on their own routes.

## Auth convention (ADR-0004)
- `registerAuth(app)` registers `@fastify/jwt` (secret = `JWT_SECRET`) and adds a **global** `preHandler` (`requireAuth`).
- `requireAuth` skips `PUBLIC_ROUTES` and unmatched routes (`request.is404` → falls through to 404, never 401); on every other route it verifies the `Authorization: Bearer` token and sets `request.authUser: { id, email }`. Tokens are header-only.
- `issueToken(user)` (exp = `JWT_TTL_HOURS`, default 24h) and `verifyToken(token)` are standalone (frozen signatures) and delegate to the single jwt instance captured at registration — no duplicate JWT setup anywhere.
- Feature modules do nothing for auth except read `request.authUser`.

## Module registration convention (ADR-0005)
`backend/src/core/modules.ts` exports `featureModules: ModuleRegistration[]` (a `{ plugin, prefix }` list), currently empty; `app.ts` loops it and calls `app.register(plugin, { prefix })`. Tolerates zero modules (mirrors A9's route-registry). To mount a module: the feature agent ships `backend/src/modules/<name>/index.ts` (default-export `FastifyPluginAsync`), then A4 appends its entry (frozen prefix table is documented inline in `modules.ts`). Adding a module never touches `package.json`.

## Config / env (ADR-0007)
`loadConfig()` reads exactly `NODE_ENV, DATABASE_URL, BACKEND_PORT, APP_BASE_URL, JWT_SECRET, JWT_TTL_HOURS, SMTP_HOST/PORT/USER/PASS/FROM`. Required (fail-fast if missing/empty): `DATABASE_URL, BACKEND_PORT, APP_BASE_URL, JWT_SECRET, SMTP_HOST, SMTP_PORT, SMTP_FROM`. Defaults: `NODE_ENV=development`, `JWT_TTL_HOURS=24`, `SMTP_USER/PASS=''`. `getConfig()` returns the cached config for other core code (A5 reads SMTP + JWT TTL).

## Changes

### 1. MF-1 — `@fastify/jwt` bump (CRITICAL, R2 MF-1 / TDR v2)
- `backend/package.json` dependencies: `@fastify/jwt` `^9.0.0` → **`10.1.0`** (exact pin per TDR "10.1.0 pinned"). Resolves `fast-jwt ^6.2.0` → **6.2.4**, patching CRITICAL GHSA-hm7r-c7qw-ghp6 (unknown `crit` header) on the sole runtime auth path.
- **NO source change required** (confirmed against `backend/src/core/auth.ts`). The plugin API surface the codebase uses is unchanged under fast-jwt 6: registration `app.register(fastifyJwt, { secret })`, decorators `host().jwt.sign/verify`, and error code `FAST_JWT_EXPIRED` mapping all remain valid. fast-jwt 6 behavioral deltas do not affect us — app tokens carry no `crit` header and use only scalar claims (`sub`/`email`/`iat`/`exp`); the "empty HMAC secret rejected" delta is now moot because the new JWT_SECRET guard (below) requires ≥32 chars. Post-bump the auth suite + `R4-repro/jwt-forge.mjs` should be re-run once A2 regenerates the lockfile (installed tree here is still 9.x).

### 3. Runtime-image hygiene (supports A2 Dockerfile prune, R4-M2 / M2)

### 5. R4-H1 (HIGH) — JWT_SECRET fail-fast guard (`backend/src/core/config.ts`)
- New `requiredJwtSecret()` reuses the existing `required('JWT_SECRET')` (missing/empty still fails fast with the ADR-0007 message), then layers two checks and throws a clear config error (same fail-fast style) if the secret is:
  1. equal to a known-weak/committed default — `WEAK_JWT_SECRETS = ['dev-only-change-me']` (the `.env.example` placeholder), OR
  2. shorter than **32 characters** (`JWT_SECRET_MIN_LENGTH`).
- `loadConfig()` now calls `requiredJwtSecret()` for `jwtSecret`. No other config key changed. This makes the default `cp .env.example .env && docker compose up` flow refuse to boot until the operator sets a strong unique secret, closing the token-forgery bypass.
