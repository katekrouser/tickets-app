## Services & ports

| Service | Image / build | Host port | Internal | Notes |
|---|---|---|---|---|
| `postgres` | `postgres:16` | none | 5432 | named volume `postgres-data`; healthcheck `pg_isready -U user -d tickets` |
| `backend` | `backend/Dockerfile` | **none (not published)** | 3000 (`expose`) | healthcheck hits `/health` via node fetch |
| `frontend` | `frontend/Dockerfile` | **`${FRONTEND_PORT}` → 80** | 80 | the ONLY published port |
| `mailhog` | `mailhog/mailhog:v1.0.1` | 8025 (UI), 1025 (SMTP) | — | `profiles: ["dev"]`, opt-in |

App origin: **http://localhost:8080**. Backend is unreachable from the host by
design (ADR-0009 §11); the browser reaches it only through nginx `/api`.

## Environment keys (ADR-0007 — exact set)

`NODE_ENV`, `DATABASE_URL`, `BACKEND_PORT`, `FRONTEND_PORT`, `APP_BASE_URL`,
`JWT_SECRET`, `JWT_TTL_HOURS`, `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`,
`SMTP_PASS`, `SMTP_FROM`. `.env.example` lists all 12 and nothing else; every
`${...}` in compose is one of these names. Real `.env` is gitignored. Postgres
init creds (`user`/`pass`/`tickets`) are literals in compose that MATCH the
default `DATABASE_URL` DSN — not env references, not secrets.

## How migrations run on boot

1. `postgres` becomes healthy (`pg_isready`).
2. `backend` `depends_on: postgres [service_healthy]`; its entrypoint
   `docker/backend-entrypoint.sh` runs **`npm run migrate`**, then
   `exec npm run start`. Migrations complete BEFORE the server accepts traffic.
3. `frontend` `depends_on: backend [service_healthy]`, then serves the SPA.

Data persists in volume `postgres-data` across restart. Fresh DB = schema +
migration metadata only, no seed data (REQUIREMENTS §9).

## nginx proxy rule (ADR-0009)

```
location /api/ { proxy_pass http://backend:3000; }   # path PRESERVED, no rewrite
location /     { try_files $uri $uri/ /index.html; }  # SPA fallback
```
Same-origin; no CORS. Backend serves only `/api/*` + `/health` + `/ready`.

2. **Prisma client generation.** `backend/Dockerfile` build stage runs

   **(SUPERSEDED by TDR v2 / R4-M2 — see "TCR-01 remediation §3":** the runtime
   stage no longer copies the whole tree; it does `npm ci --omit=dev` and copies

   now **prod** deps, so migrate-at-boot works without shipping dev/test tooling.)

Backend npm scripts (A4): `migrate` = `prisma migrate deploy`, `start` =
`node dist/server.js`. Entrypoint runs `npm run migrate` then `npm run start`.

## Expected `docker compose up --build` sequence

1. `postgres` starts → becomes healthy (`pg_isready`).
2. Images build: backend runs `npm ci` → `prisma generate` → `tsc` build;
   frontend runs `npm ci` → `vite build` → assets copied into nginx.
3. `backend` waits for postgres-healthy, entrypoint runs `prisma migrate deploy`
   (applies schema to the fresh DB), then starts Fastify on 3000 serving
   `/api/*` + `/health` + `/ready`; healthcheck polls `/health`.
4. `frontend` waits for backend-healthy, then nginx serves the SPA and proxies
   `/api/*` → `backend:3000`. Open http://localhost:8080.

## Watch-outs during a live run

- **First build is slow** (two full `npm ci` installs + Prisma engine download +
  Vite build). This is normal; subsequent builds use layer cache.
- **Fresh DB has no data** by design (§9). Sign up via the UI to create data;
  verification email goes to Mailhog only if started with `--profile dev`
  (UI at http://localhost:8025). Without the dev profile, `SMTP_HOST=mailhog`
  has no target and verification emails will fail to send.
- Backend is intentionally **not** reachable from the host; hit the app only via
  http://localhost:8080 (nginx). `/health` and `/ready` are container-internal.
- If `.env` is missing, `cp .env.example .env` first — compose interpolates
  `${...}` from it.

### 2. R4-H1 — committed weak `JWT_SECRET` removed; zero-config preserved
- `.env.example`: `JWT_SECRET=dev-only-change-me` → **`JWT_SECRET=`** (empty) with
  a comment (">=32 chars; leave unset for an auto-generated ephemeral dev secret;
  MUST set in production").
- `docker-compose.yml`: `JWT_SECRET: ${JWT_SECRET}` → **`${JWT_SECRET:-}`** so an
  unset var doesn't hard-warn.
- `docker/backend-entrypoint.sh`: BEFORE migrate/start, if `$JWT_SECRET` is empty,
  `<32` chars, or `= dev-only-change-me`, generate `crypto.randomBytes(48).hex`
  via node, `export` it, and print a one-line ephemeral-secret notice. This keeps
  `docker compose up` zero-config yet never boots on the publicly-known key, and
  satisfies A4's boot guard (fail-fast <32 chars / placeholder). README documents
  the behavior + production requirement.
