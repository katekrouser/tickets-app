## Import alias (single-valued, ADR-0008)

`@` → `frontend/src`, configured in BOTH `tsconfig.json` (`paths: { "@/*": ["src/*"] }`) and
`vite.config.ts` (`resolve.alias['@'] = <repo>/frontend/src`). The only import forms are `@/…` (foundation)
and each feature's published `@/features/<name>/routes` module. Deep-importing another feature's internals is
forbidden.

Consumers import ONLY these; do NOT instantiate a second API client or a second auth context.

## Route-registry — how a feature contributes routes (ADR-0008)

Each feature exports `routes: RouteObject[]` from **`@/features/<name>/routes.tsx`**. `@/app/router` AUTO-DISCOVERS
them with Vite `import.meta.glob('../features/*/routes.tsx', { eager: true, import: 'routes' })` and composes them
as children of the shared `<AppShell>` layout. **A feature registers screens by creating its `routes.tsx` — with
NO edit to `@/app`.** Features do not call any runtime registrar.

```ts
// src/features/board/routes.tsx  (A12, example)
import type { RouteObject } from 'react-router-dom';
import { RequireAuth } from '@/lib/auth';
export const routes: RouteObject[] = [
  { path: '/board', element: <RequireAuth><BoardPage /></RequireAuth> },
];
```

- Protected screens wrap their `element` in `<RequireAuth>` (imported from `@/lib/auth`).
- Public screens (A10's `/login`, `/signup`) are plain routes (no `RequireAuth`). They render inside `<AppShell>`;
  the header's user menu auto-hides when there is no user, so a bare-header login page is fine.
- **A13** additionally exports `ticketRoutes` from `@/features/tickets/routes.tsx` (`create()`, `detail(id)`,
  `edit(id)`) per ADR-0008; **A12** navigates via that constant. That is A13→A12's boundary, not A9's.

### Landing route (`/`) behavior
The index route wraps a redirect in the guard: `<RequireAuth><Navigate to="/board" replace /></RequireAuth>`.
- **Authenticated** `/` → redirect to `/board` (A12's board feature).
- **Unauthenticated** `/` → `RequireAuth` redirects to `/login` (A10's auth feature) — the guard wraps the
  redirect, so the login redirect does not depend on `/board`'s own guard.
- Unknown paths render a NotFound `EmptyState`.

The `/board`, `/login`, `/signup` targets are feature-owned and are picked up automatically by the glob. With
zero features the glob resolves to `{}`; the app still composes and runs (targets arrive once features register).

## API transport / Vite proxy (ADR-0009 + A1 Prism advisory)

- `@/lib/api` sets `baseUrl: '/api'` — relative, unchanged between dev and prod (ADR-0009 §4).
- `vite.config.ts` proxies `/api` → **exactly** `http://localhost:3000` (ADR-0009 §10) — the single target that
  serves both the real backend and the Prism mock in dev.
- **Path handling switch (`VITE_USE_MOCK`)** — per A1's handoff advisory that Prism 5 ignores the OpenAPI `servers`
  base path and serves operations at ROOT:
  - **Default (real backend):** path is PRESERVED. `/api/teams` → `http://localhost:3000/api/teams`.
  - **`VITE_USE_MOCK=1` (or `true`):** a `rewrite` strips the leading `/api`. `/api/teams` → `http://localhost:3000/teams`.
  - The generated `paths` keys stay root-relative, exactly what `openapi-fetch` + `baseUrl:'/api'` expects — no
    type/client change is needed for either mode.
- **Authorization header (Phase 2).** `@/lib/api` registers an `openapi-fetch` request middleware
  (`api.use({ onRequest })`) that reads the current token via `getStoredToken()` and sets
  `Authorization: Bearer <token>` on every request; when signed out it sets nothing. This satisfies the real
  backend's global `requireAuth` (ADR-0004) and is a no-op against the Prism mock. `baseUrl` remains `'/api'`.

Dev commands:
```bash
# against the real backend (default — path preserved)
npm run dev --workspace frontend

# against the Prism mock (strips /api).  Start the mock first:  npm run mock   (repo root → Prism on :3000)
VITE_USE_MOCK=1 npm run dev --workspace frontend
```

## Notes / advisories for downstream agents

- **`src/features/**` is unowned by A9** and does not yet exist — A10–A13 create their own feature folders. The
  router glob picks up any `src/features/<name>/routes.tsx` automatically.
- **Token storage is NOT the data system of record** (TDR §3 / REQUIREMENTS §9). `@/lib/auth` persists only the
  bearer token (localStorage key `tickets.auth.token`) so a reload keeps the session; `user` is derived from the
  JWT claims (`sub`→id, `email`). All domain data comes from the API via `@/lib/api` + TanStack Query.
- `login(token)` takes only the token (per the frozen signature). A10 calls it with the token from `POST /auth/login`.
- A single `QueryClient` is created in `@/app/providers`; features add their own `useQuery`/`useMutation` — they must
  not create another client or provider.
- Vite prints a cosmetic "CJS build of Vite's Node API is deprecated" warning because `vite.config.ts` uses
  `__dirname` (loaded as CJS). Harmless; build and dev are unaffected.

## Phase 2 integration changes (A9)

Two foundation fixes flagged by A12/A13, applied within `frontend/src/{app,lib,components}`:

1. **Bearer auth on every request.** `@/lib/api` now registers an `openapi-fetch` request middleware that
   attaches `Authorization: Bearer <token>` from `getStoredToken()` (skips when signed out). Added
   `getStoredToken()` to `@/lib/auth` as the single reader of the storage key (`tickets.auth.token`). `baseUrl`
   is unchanged (`'/api'`, ADR-0009). No effect on the Prism mock; makes authenticated calls work against the
   real backend (ADR-0004 global `requireAuth`).
2. **Landing route.** `/` now redirects authenticated users to `/board`; the redirect is wrapped in
   `RequireAuth`, so unauthenticated users still route to `/login`. Chosen the redirect (minimal surface) over a
   persistent header nav. Public auth routes (`/login`, `/signup`, …) are unaffected.
